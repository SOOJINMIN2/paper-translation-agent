# 논문번역 에이전트 설계서

## 1. 프로젝트 개요

### 1.1 핵심 가치 명제
기존 DeepL·Google Translate는 논문의 **구조적 무결성**을 보존하지 못한다.
두 단 레이아웃이 뒤섞이거나, 수식·그림·표가 번역 중 소실되거나, 전문 용어가 맥락 없이 번역되는 문제를 해결한다.

**이 에이전트의 차별점:**
- 섹션·수식·그림·표 구조를 원문 그대로 유지한 채 번역
- 저자의 문체(어조·문장 길이·서술 패턴)까지 한국어로 승계
- 전문 용어를 원어 병기로 처리하여 학문적 정밀도 유지
- 논문 간 용어 일관성을 위한 지속적 용어집 관리

### 1.2 사용 컨텍스트
- **사용자**: 단일 사용자 (개인 연구 보조 도구)
- **대상 분야**: 자연과학 (물리·화학 중심, 수식 집약적)
- **대상 언어**: 영어로 작성된 논문 (비영어 논문은 지원 범위 외)
- **워크플로우**: 논문 PDF 입력 → 15~30분 대기 → 번역된 Markdown/HTML 파일로 읽기
- **실행 환경**: Claude Code Agent 대화창에서 PDF 경로 입력으로 실행 (슬래시 커맨드 또는 자연어 인식)

## 2. 시스템 아키텍처

### 2.1 멀티 에이전트 파이프라인

```
[사용자] PDF 경로 입력
    ↓
[Orchestrator Agent]  ← 전체 파이프라인 제어 / 에러 핸들링 / 재개 여부 판단
    ├─→ [Parsing Agent]         PDF → 구조화 JSON (Marker 신뢰도 기반 폴백)
    ├─→ [Style Analysis Agent]  원문 전문 용어 밀도·수식 집약도 분석 → 번역 지침 생성  [NEW]
    ├─→ [Priority Agent]        섹션 중요도 분석 → 배정 결과 파일 저장 → 사용자 확인 후 시작
    ├─→ [Translation Agent]     섹션별 번역 (우선순위 기반 모델 선택, 청크 내 용어 일관성 관리)
    ├─→ [Reviewer Agent]        번역 품질 검증 → 이슈 시 자동 재번역 시도
    └─→ [Reconstruction Agent]  Markdown + HTML 생성 (부분 재실행 지원)
    ↓
[출력] paper_kr.md + paper_kr.html + paper_figures/ (PDF와 같은 폴더)
```

### 2.2 에이전트 간 데이터 흐름

모든 에이전트는 **중간 JSON 파일**을 통해 데이터를 주고받는다.
중간 파일이 이미 존재하면 해당 단계를 건너뛰어 **부분 재실행**이 가능하다.

```
paper.pdf
  → paper_parsed.json        (Parsing Agent 출력)
  → paper_style.json         (Style Analysis Agent 출력)  [NEW]
  → paper_priority.json      (Priority Agent 출력 → 사용자가 직접 편집 가능)
  → paper_translated.json    (Translation Agent 출력)
  → paper_reviewed.json      (Reviewer Agent 출력)
  → paper_kr.md              (Reconstruction Agent 최종 출력)
  → paper_kr.html            (Reconstruction Agent 최종 출력)
  → paper_figures/           (Parsing Agent가 추출한 그림 파일 폴더)  [NEW]
  → paper_review_report.json (이슈 있을 때만 생성)
```

## 3. 에이전트별 역할 명세

### 3.1 Orchestrator Agent
- 전체 파이프라인 순서 제어
- **시작 시 재개 여부 판단**: 중간 JSON 파일이 존재하면 사용자에게 "처음부터 시작 / 이어서 시작" 선택 질문
- 각 단계 성공/실패 감지 및 로깅
- 에러 발생 시 폴백 전략 실행
- 콘솔 진행 상황 실시간 출력:
  - `[3/12 섹션] Introduction 번역 완료 (1분 23초 경과, 잔여 ~8분)`
  - 섹션 완료 시마다 처리 페이지 수 + 소요 시간 표시

**재개 흐름:**
```
중간 파일 감지
  → "이전 작업이 있습니다. 이어서 시작할까요? [Y/n]"
  → Y: 가장 마지막 완료 단계 이후부터 실행
  → n: 모든 중간 파일 삭제 후 처음부터 실행
```

### 3.2 Parsing Agent

**주 전략: Marker 라이브러리 (ML 기반 PDF→Markdown)**
- `marker-pdf` 라이브러리로 PDF를 구조화된 Markdown으로 변환
- 두 단(two-column) 레이아웃 자동 인식 및 읽기 순서 복원
- 수식 추출: LaTeX 블록(`$...$`, `$$...$$`)으로 보존
- 그림·표 감지: 위치 메타데이터 포함하여 기록
- **그림 파일 저장**: `paper_figures/` 폴더를 생성하여 PNG로 추출 저장

**폴백 트리거: Marker 신뢰도 점수**
- Marker가 페이지/섹션 단위로 신뢰도 점수를 제공함
- 임계값(threshold) 미만 시 해당 페이지만 Vision으로 자동 전환
- 전체 신뢰도가 심각하게 낮으면 논문 전체를 Vision으로 폴백
- **DRM 보호 PDF**: Marker 실패 시 자동으로 Vision 폴백 적용 (OCR 방식)

**폴백 전략: Claude Vision**
- PDF를 페이지 이미지로 변환 후 Claude API vision 입력
- 텍스트·수식·레이아웃 구조를 OCR로 인식
- Vision 폴백 시에도 `paper_figures/`에 그림 파일 저장 (페이지 스크린샷 크롭)

**출력 JSON 스키마:**
```json
{
  "paper_id": "paper_filename",
  "sections": [
    {
      "id": "abstract",
      "title": "Abstract",
      "content": "원문 텍스트 (수식 플레이스홀더 포함)",
      "level": 1,
      "has_equations": true,
      "equations": ["$$E = mc^2$$"],
      "figures": [
        { "id": "fig1", "image_path": "paper_figures/fig1.png", "caption": "Figure 1: ..." }
      ],
      "tables": [
        { "id": "tab1", "caption": "Table 1: ...", "content": "마크다운 표", "is_text_heavy": true }
      ],
      "references_inline": ["[1]", "[2]"]
    }
  ],
  "references": [
    { "id": "[1]", "text": "Author, Title, Journal, Year" }
  ],
  "parse_method": "marker",
  "parse_issues": []
}
```

### 3.3 Style Analysis Agent (신규)

번역 전 원문의 **전문 용어 밀도 및 수식 집약도**를 분석하여 Translation Agent에 번역 지침으로 주입한다.

**분석 항목:**
- 수식 집약도: 전체 텍스트 대비 수식 블록 비율 (수식 주도형 vs 내러티브 주도형 분류)
- 전문 용어 밀도: 섹션당 신규 전문 용어 등장 빈도
- 분야 성격 분류: `equation-heavy` | `narrative-heavy` | `mixed`

**출력 JSON 스키마 (paper_style.json):**
```json
{
  "paper_id": "paper_filename",
  "style_type": "equation-heavy",
  "equation_ratio": 0.38,
  "term_density": "high",
  "translation_guidance": "이 논문은 수식 집약도가 높고 기술적 용어 밀도가 높습니다. 수식 주변 설명 문장은 간결하게 번역하되, 수식과의 연결성을 유지하십시오."
}
```

### 3.4 Priority Agent

Abstract를 먼저 분석하여 논문의 핵심 연구 주제를 파악한 뒤,
각 섹션이 해당 연구에서 갖는 중요도를 내용 기반으로 판단하여 `high | medium | low`로 배정한다.

**배정 완료 후 사용자 확인 흐름:**
1. `paper_priority.json` 파일에 배정 결과 저장
2. 콘솔에 배정 결과 목록 출력
3. "수정 사항이 있으면 paper_priority.json을 직접 편집 후 Enter를 누르세요." 대기
4. 사용자가 JSON을 편집하거나 그냥 Enter를 누르면 Translation Agent 시작

**자연과학 논문 기준 기본값:**

| 섹션 유형 | 기본 우선순위 | 비고 |
|---|---|---|
| Abstract | high | 항상 고품질 필수 |
| Introduction | high | 맥락 이해의 핵심 |
| Methods / Experimental | high | 재현성에 중요 |
| Results | high | 핵심 발견 기술 |
| Discussion / Conclusion | high | 해석과 의의 |
| Supplementary Material | low | 세부 보조 자료 (사용자 재지정 가능) |
| Appendix | low | 부록 |
| References | medium | 구조 보존 필요 |

### 3.5 Translation Agent

- 섹션 단위로 분할하여 컨텍스트 윈도우 관리
- 우선순위에 따른 모델 선택 (high → claude-sonnet-4-6 이상, low/medium → claude-haiku-4-5) ← API 비용 최적화
- 번역 시작 전 용어집 파일(`glossary.json`)의 해당 분야 항목 + `no_translate.txt` 목록을 로드하여 프롬프트에 주입
- Style Analysis Agent의 `translation_guidance`를 시스템 프롬프트에 포함

**긴 섹션 청킹 전략:**
- 섹션이 컨텍스트 윈도우를 초과하는 경우 서브섹션(3.1, 3.2 등) 단위로 분할
- 각 청크 번역 시 프롬프트에 주입:
  - 전역 용어집 항목
  - 현재 논문에서 직전 청크까지 확정된 신규 용어 번역 목록
- 청크 간 용어 일관성은 위 누적 용어 목록으로 보장

**번역 7대 원칙:**

1. **유창성 우선**: 자연스러운 한국어 산문이 목표. 직역이 어색할 경우 적극적으로 의역
2. **저자 문체 승계**: Style Analysis Agent의 분석 결과를 반영. 수식 집약형 논문은 수식 주변 설명을 간결하게 유지
3. **전문 용어**: `한국어 번역(원어)` 형식으로 초출 시 병기
   예: `경사 하강법(gradient descent)`, `파동 함수(wave function)`
4. **번역 불가 용어**: `no_translate.txt`에 등록된 용어는 원어 그대로 유지
   예: `laser`, `transformer`, `plasma`
5. **수식**: 번역하지 않음. `$...$`, `$$...$$` LaTeX 블록 원문 그대로 보존
6. **그림 캡션**: 번역. 본문 내 `Figure N` → `그림 N`으로 통일
7. **표 내용**: 모든 표(데이터 표 포함)의 텍스트 내용을 번역하되 원어를 병기
   예: 헤더 `Activation Energy` → `활성화 에너지(Activation Energy)`
8. **참고문헌**: 논문 제목은 번역, 저자명·학술지명은 원어 유지

**Translation Agent 시스템 프롬프트 핵심 지침:**
```
당신은 자연과학(물리·화학) 논문 전문 번역가입니다.

[문체 지침]
{style_analysis.translation_guidance}

[번역 불가 용어 목록]
다음 용어는 원어 그대로 유지하십시오: {no_translate_list}

[용어집]
{glossary_terms}

[이번 청크까지 확정된 신규 용어]
{chunk_term_log}

[필수 규칙]
- 수식 블록($...$, $$...$$)은 단 한 글자도 수정하지 않는다
- 전문 용어 첫 등장 시 '한국어(원어)' 형식으로 병기
- 이전 섹션에서 확정한 용어 번역과 100% 일관성 유지
- 표의 텍스트 내용도 번역하되 원어 병기

[번역 목표]
저자의 의도와 어조를 살리면서도 한국어 독자가 자연스럽게 읽을 수 있는 번역.
유창성이 충실도보다 우선이지만, 과학적 정확성은 절대 희생하지 않는다.
```

### 3.6 Reviewer Agent

번역 완료 후 다음 항목을 검증하고 이슈가 발견되면 자동 재번역을 시도한다.

**검증 체크리스트:**

1. **전문 용어 일관성**: 동일한 원어 용어가 논문 전체에서 동일한 한국어 번역으로 사용되었는지
2. **누락 단락 탐지**: 원문 섹션 수와 번역문 섹션 수 1:1 매핑 검증
3. **반복 단락 탐지**: 동일하거나 매우 유사한 문장이 두 번 이상 번역된 경우 감지
4. **섹션 구조 매핑**: 원문의 Abstract, Introduction, Methods 등이 번역문에 빠짐없이 대응되는지

**이슈 대응 전략:**

| 이슈 심각도 | 대응 |
|---|---|
| Critical (누락 단락, 섹션 통째 누락) | 해당 섹션 Translation Agent로 자동 재번역 시도 (최대 2회) |
| 재번역 후에도 실패 | 원문과 번역문을 병렬 표시하여 사용자가 직접 판단할 수 있도록 출력 |
| Warning (용어 불일치, 반복 단락) | `paper_review_report.json`에 기록 후 계속 진행 |

**출력:** 이슈 발견 시 콘솔 경고 + `paper_review_report.json` 생성. 이슈가 없으면 리포트 파일 미생성.

### 3.7 Reconstruction Agent

검증된 번역 JSON에서 최종 파일 두 가지를 생성한다.
**용어집 충돌 수정 후 이 에이전트만 단독 재실행하여 출력 파일을 업데이트할 수 있다.**

**Markdown 출력 (`paper_kr.md`):**
- 섹션 제목 계층(`# ## ###`) 구조 보존
- 수식: `$...$`, `$$...$$` LaTeX 블록 그대로 유지 (Obsidian에서 렌더링)
- 그림: `![그림 N](paper_figures/figN.png)` 상대 경로 + 번역 캡션
- 표: Markdown 표 형식으로 재구성, 셀 내용 번역 + 원어 병기
- 참고문헌: 번역된 제목 + 원어 저자명/학술지명

**HTML 출력 (`paper_kr.html`):**
- MathJax CDN으로 LaTeX 수식 실시간 렌더링
- **Sidebar TOC (목차)**: 섹션·서브섹션을 빠르게 탐색할 수 있는 고정 사이드바 목차
- 섹션 헤딩에 앵커 ID 자동 생성 (TOC 링크 대상)
- 그림 이미지 상대 경로로 `<img>` 태그 삽입 (paper_figures/ 기준)
- 참고문헌 번호 앵커 링크 (`[1]` 클릭 → 해당 ref로 이동)
- 인라인 인용 hover tooltip (마우스 오버 시 논문 제목 표시)
- 단일 컬럼 레이아웃 (두 단 CSS 재현 미지원, 가독성과 탐색성 우선)

## 4. 용어집 관리 시스템

### 4.1 파일 구조 및 위치

**경로 설정:** `CLAUDE.md`에 `GLOSSARY_PATH` 환경변수로 지정 (Windows 환경 대응)
```
# CLAUDE.md 예시
GLOSSARY_PATH=C:\Users\username\papers\glossary
```

**파일 구성:**
```
$GLOSSARY_PATH/
  glossary.json        ← 분야별 용어 번역 (영구 공유)
  no_translate.txt     ← 번역하지 않는 용어 목록 (영구 공유)
```

**glossary.json 구조:**
```json
{
  "physics": {
    "electron spin": "전자 스핀",
    "wave function": "파동 함수",
    "Hamiltonian": "해밀토니안",
    "eigenvalue": "고유값",
    "entanglement": "얽힘"
  },
  "chemistry": {
    "reaction rate": "반응 속도",
    "activation energy": "활성화 에너지",
    "catalyst": "촉매",
    "valence electron": "원자가 전자"
  }
}
```

**no_translate.txt 구조 (1행당 1용어):**
```
laser
transformer
plasma
photon
qubit
```

### 4.2 용어집 충돌 해결 프로세스

번역 완료 후 Reviewer가 **포스트 프로세싱** 단계에서 충돌을 감지하여 사용자에게 일괄 제시한다.

**충돌 정의**: 기존 glossary.json의 용어 번역이 현재 논문 맥락에서 다르게 번역된 경우
예: `state` → 기존 용어집은 `상태`, 현재 논문에서는 `준위`로 번역

**충돌 해결 흐름:**
```
번역 완료
  → Reviewer가 용어집-번역문 간 충돌 목록 추출
  → 콘솔에 충돌 목록 출력:
      [충돌] 'state': 용어집='상태', 이번 번역='준위'
      → 어느 쪽을 사용할까요? [1=용어집, 2=이번 번역, 3=스킵]
  → 사용자가 항목별로 선택
  → 선택 결과를 paper_translated.json에 반영
  → Reconstruction Agent만 재실행하여 md/html 업데이트
  → 선택한 번역을 glossary.json에 자동 추가 (신규 용어) 또는 유지 (기존 용어 우선 시)
```

### 4.3 운용 방식
1. **번역 시작 전**: Orchestrator가 해당 분야 용어집 + no_translate.txt 로드 → 각 Agent 프롬프트에 삽입
2. **번역 완료 후**: 충돌 해결 프로세스 실행, 신규 용어는 자동으로 용어집에 추가
3. **다음 논문**: 동일 분야 번역 시 기존 용어집을 재사용하여 논문 간 일관성 유지
4. **용어집 없음**: 빈 용어집 + 빈 no_translate.txt로 자동 시작, 신규 생성

## 5. 파일 입출력 명세

### 5.1 입력
```
사용자가 Claude Code 대화창에 입력:
"이 논문 번역해줘: /path/to/paper.pdf"
또는 슬래시 커맨드: /translate-paper /path/to/paper.pdf

Supplementary Material이 별도 PDF인 경우:
"이 논문 번역해줘: /path/to/paper.pdf /path/to/supplementary.pdf"
→ 주 논문과 동일한 파이프라인으로 연속 처리
→ 출력 형식(합친 파일 vs 분리 파일)은 실행 시 사용자가 지정
```

### 5.2 중간 파일 (같은 폴더, 번역 완료 후 삭제 가능)
```
paper_parsed.json
paper_style.json            ← 문체 분석 결과 (Style Analysis Agent)
paper_priority.json         ← 섹션 우선순위 (사용자가 직접 편집 가능)
paper_translated.json
paper_reviewed.json
paper_review_report.json    ← 이슈 있을 때만 생성
```

### 5.3 최종 출력 (같은 폴더)
```
paper_kr.md           ← Obsidian / VS Code에서 읽기 (수식 렌더링 포함)
paper_kr.html         ← 브라우저에서 MathJax 렌더링 + Sidebar TOC
paper_figures/        ← 논문에서 추출한 그림 파일 (PNG)
  fig1.png
  fig2.png
  ...
```

### 5.4 영구 설정 파일 ($GLOSSARY_PATH/)
```
glossary.json         ← 분야별 용어 번역 (논문 간 공유)
no_translate.txt      ← 원어 유지 용어 목록 (논문 간 공유)
```

## 6. 오류 처리 전략

| 오류 상황 | 대응 방식 |
|---|---|
| Marker 신뢰도 점수 임계값 미만 (페이지 단위) | 해당 페이지만 Claude Vision으로 자동 전환 |
| Marker 전체 실패 | 논문 전체를 Claude Vision으로 폴백, 콘솔 알림 |
| DRM 보호 PDF (텍스트 추출 불가) | Claude Vision으로 자동 폴백 (OCR 방식) |
| API 타임아웃 | 해당 섹션 최대 3회 재시도 후 스킵 + 로그 기록 |
| Reviewer Critical 이슈 탐지 | 해당 섹션 자동 재번역 (최대 2회) |
| 재번역 후에도 Critical 이슈 | 원문 + 착오 번역문을 병렬 표시, 사용자 판단에 맡김 |
| Reviewer Warning 이슈 탐지 | `paper_review_report.json` 기록 후 계속 진행 |
| 용어집 파일 없음 | 빈 용어집으로 자동 시작, 신규 생성 |
| 중간 파일 손상 | 해당 단계부터 재시작 |
| 실행 중 사용자 중단 (Ctrl+C) | 중간 JSON 보존, 다음 실행 시 이어서/처음부터 선택 질문 |

## 8. 구현 로드맵

### Phase 1: 핵심 파이프라인 MVP
- [ ] Parsing Agent: Marker 연동 + 구조화 JSON 출력 + paper_figures/ 생성
- [ ] Translation Agent: 섹션 단위 번역 + 용어집 주입 + 표 전체 번역(원어 병기)

- [ ] Style Analysis Agent: 수식 집약도 및 전문 용어 밀도 분석
- [ ] Priority Agent: Abstract 분석 기반 섹션 우선순위 자동 결정 + 사용자 확인 단계
- [ ] Marker 신뢰도 점수 기반 페이지 단위 Vision 폴백
- [ ] Reviewer Agent: 용어 일관성 + 누락/반복 단락 탐지 + 자동 재번역
- [ ] HTML 출력: MathJax + Sidebar TOC + 앵커 링크 + hover tooltip

- [ ] Supplementary Material 별도 PDF 연속 처리 지원
- [ ] Reviewer 이슈 리포트 포맷 정제 + 원문-번역 병렬 표시

## 9. 핵심 설계 트레이드오프

| 결정 항목 | 최종 선택 | 포기한 것 |
|---|---|---|
| 유창성 vs 충실도 | **유창성 우선 (의역)** | 원문과 1:1 직접 대조의 어려움 |
| 단일 vs 멀티 에이전트 | **멀티 에이전트** | 구현 복잡도 증가 |
| 즉시 응답 vs 품질 | **15~30분 수용** | 즉각적인 피드백 없음 |
| 수식 번역 vs 보존 | **보존 (번역 제외)** | 수식 주변 설명과의 흐름 단절 가능성 |
| 저자 문체 vs 균일 가독성 | **저자 문체 승계 (수식 집약도 반영)** | 복잡한 원문은 한국어도 복잡하게 읽힘 |
| 표 원문 유지 vs 전체 번역 | **전체 번역 + 원어 병기** | 번역 비용·시간 증가 |
| HTML 두 단 레이아웃 vs 탐색성 | **Sidebar TOC 우선 (단일 컬럼)** | PDF 원문과 시각적 일치 포기 |
| 용어집 일관성 vs 맥락 적합성 | **포스트 프로세싱으로 사용자가 결정** | 완전 자동화의 편의성 |
| 비용 균등 vs 비용 최적화 | **우선순위 연동 모델 선택** | 저우선순위 섹션 품질 일부 희생 |
